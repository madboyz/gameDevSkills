# 开发计划维护技能

## 核心目标

在每次开发任务中，为当前变更留下可交接的计划记录。计划只记录本次任务的开发进度、已完成内容、缺失内容、验证结果和下次入口，不替代 `doc/` 下的专题设计文档。

## 使用流程

1. 在开始修改前识别本次任务的主模块。
2. 根据主模块确定 `doc/plan/<模块分类>/` 目录。
3. 每次任务新建一个独立计划文件，不复用旧任务文件。
4. 开发过程中持续更新计划：完成一项记录到“已完成”，发现未处理事项记录到“未完成 / 缺失内容”。
5. 任务结束前补齐“已验证”和“下次接手建议”。
6. 如果本次变更触发项目文档维护规则，仍需同步对应专题文档；`doc/plan` 只作为进度与交接记录。

## 路径分类规则

优先按主要修改文件所属业务路径映射：

- `assets/scripts/logic/scene/map/*` -> `doc/plan/scene/map/`
- `assets/scripts/logic/scene/team/*` -> `doc/plan/scene/team/`
- `assets/scripts/logic/scene/hub/*` -> `doc/plan/scene/hub/`
- `assets/scripts/logic/uilayer/<module>/*` -> `doc/plan/uilayer/<module>/`
- `assets/scripts/framework/<module>/*` -> `doc/plan/framework/<module>/`
- `assets/scripts/gamestart/*` -> `doc/plan/gamestart/`
- 跨多个模块时，选择本次任务的主行为模块；其余模块写入“当前涉及文件”。

若没有明确匹配项，使用最接近的业务目录名生成分类，例如 `assets/scripts/logic/foo/bar/*` 对应 `doc/plan/foo/bar/`。目录名保持小写；已有中文计划文件名可以继续使用中文。

## 文件命名

- 每次任务新建文件，格式为：`<中文任务短标题>计划.md`。
- 标题应能表达行为目标，例如：`队伍移动路径修复计划.md`、`世界地图寻路对齐计划.md`。
- 若同名文件已存在，在标题后追加日期或序号，例如：`队伍移动路径修复计划-20260522.md`。

## 计划模板

```markdown
# <任务标题>

## 背景与目标

说明本次任务为什么要做、目标是什么、对齐来源或关键约束是什么。

## 当前涉及文件

- `<路径>`：说明本次修改或核对的职责。

## 已完成

- 已完成的实现、修复、接口调整、文档同步或验证准备。

## 未完成 / 缺失内容

- [ ] 尚未处理的内容。
  - 为什么未完成：说明原因。
  - 下一步入口：给出具体文件、函数、命令或验证场景。

## 已验证

- 已执行的 lint、测试、静态搜索或人工核对结果。
- 若无法验证，写明阻塞原因和建议验证命令。

## 下次接手建议

1. 给出下一位开发者最该先看的入口。
2. 标明容易混淆的旧实现、业务分支或运行时约束。
```

## 写作要求

- 使用简体中文。
- 内容以事实和下一步动作为主，不写泛泛总结。
- “已完成”和“未完成 / 缺失内容”必须同时存在；没有缺失内容时写“暂无已知缺失”。
- 路径使用仓库相对路径。
- 对齐旧工程时写明具体参考文件，不把相近但不同的旧实现混为一套规则。
- 计划文件是当前任务交接记录，不反向修改历史计划，除非用户明确要求整理历史。
